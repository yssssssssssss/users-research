# AI 用研分析系统整体方案

> 文档日期：2026-03-18  
> 适用范围：`users-research` 项目总体设计、前后端实现、模型接入、状态治理、输出门禁、研发拆解  
> 约束前提：**保留**多 Agent、多模型 fan-out、Vision MoE、Persona Sandbox、多输出路由能力；模型调用能力基于现有 `modelClients.ts`；现有 URL/Key 配置来源于 `.env.local`

---

## 1. 文档目标

本文档用于沉淀一版可直接进入研发拆解的整体方案，覆盖：

- 总体架构与控制拓扑
- 多 Agent / 多模型 / Vision / Persona 的职责边界
- 前端展示与用户交互方案
- 后端服务、状态机、数据模型与 API 契约
- 输出门禁与证据治理
- 观测、降级、恢复与研发实施建议

本文档的目标不是削减能力，而是在**保留高级能力**的前提下，把系统从“模型集合”提升为“可追溯、可门禁、可审阅”的 AI 用研分析工作台。

---

## 2. 核心结论

### 2.1 方案定位

系统应被定义为：

> **研究控制面 + 模型能力面 + 证据状态面 + 工作台前端**

而不是：

> 单一聊天机器人 / 简单 Prompt 编排工具 / 无状态模型调用面板

### 2.2 设计原则

1. **保留高级能力，但必须受控**
   - 保留多 Agent、多模型 fan-out、Vision MoE、Persona Sandbox、多输出路由
   - 但所有高级能力都必须进入统一编排、证据治理与输出门禁

2. **复用现有模型能力资产**
   - 继续使用 `modelClients.ts` 作为模型能力适配层
   - 不推倒重写模型调用逻辑

3. **正式环境必须服务端收口**
   - 当前 `.env.local` 中存在 `VITE_*` 形式的模型配置项
   - 原型阶段可使用，但正式系统必须改为服务端读取并代理调用

4. **前端必须是“研究工作台”**
   - 不能只有聊天输入框
   - 必须承载任务、证据、视觉、Persona、报告、审核、观测

5. **输出由 Gate 决定，不由模型自由发挥**
   - 报告类型、报告级别、可见内容必须由门禁规则统一裁决

## 4. 总体架构

### 4.1 系统总览

```text
[Web 前端工作台]
    |
    v
[BFF / API Gateway]
    |
    +-----------------------------+
    |                             |
    v                             v
[Research Orchestrator]      [Task / Review / Auth APIs]
    |
    +--> Problem Decomposer
    +--> Evidence Retrieval & Alignment
    +--> Vision MoE
    +--> Persona Sandbox
    +--> Judgment Synthesizer
    +--> Output Router / Gate Evaluator
    |
    v
[State Store + Evidence Store + Model Run Logs]
    |
    +--> PostgreSQL / pgvector
    +--> Redis / Queue
    +--> JD OSS
    +--> 内部知识库 / 外部搜索
    +--> Model Gateway（复用 modelClients.ts）
```

### 4.2 三层控制拓扑

#### 1）控制面

负责“系统如何运行”：

- 任务编排
- Agent 路由
- fan-out/fan-in
- 降级策略
- 输出门禁
- 审核流与回退

#### 2）数据面

负责“系统在处理什么”：

- 用户问题
- 上传文档
- 设计图
- 证据数据
- 模型输出
- 报告内容

#### 3）状态面

负责“系统已经知道什么事实”：

- 任务状态
- 子问题
- 证据池
- 冲突组
- Vision 发现
- Persona 发现
- 模型运行记录
- 报告版本
- 审核动作
- Checkpoint

## 5. 控制合同（Control Contract）

### Primary Setpoint

构建一个在保留多 Agent、多模型、Vision、Persona 能力前提下，仍然**可追溯、可门禁、可审阅**的 AI 用研分析系统。

### Acceptance

系统至少应满足：

- 支持用户输入研究问题、上传文档、上传设计图
- 支持问题拆解、证据检索、设计评估、Persona 压力测试、综合判断
- 支持判断卡片 / 轻量报告 / 证据型报告 / 设计预评估 / 假设包输出
- 能区分事实、推断、模拟、缺口
- 每条关键结论可回溯到证据来源

### Guardrail Metrics

- 不泄露模型 API Key
- 不把 Persona 模拟内容当真实用户证据
- 不把 Vision 结果直接升格为 T1 事实
- 不允许低证据生成高置信报告
- 不允许输出门禁逻辑分散在前后端多个事实源中

### Recovery Target

若支路失败，系统应可：

- 降级继续运行主链
- 输出受限版结果
- 保留失败证据与观测日志
- 允许人工复核与重跑

## 6. 核心模块设计

### 6.1 Problem Decomposer

#### 职责

- 将模糊问题拆成 2~5 个子问题
- 输出研究维度
- 将输入归类为文本 / 设计 / 混合

#### 输入

- 用户文本问题
- 上传文档摘要
- 设计图上下文

#### 输出

- `sub_questions`
- `research_axes`
- `task_type`

---

### 6.2 Evidence Retrieval & Alignment

#### 职责

- 内部资料检索
- 外部检索
- 证据去重
- 冲突识别
- Tier 标注
- RQ 评分

#### 输出

- `evidence_pool`
- `evidence_conflicts`
- `rq_level`
- `evidence_gap`

---

### 6.3 Vision MoE

#### 保留原则

保留多模型视觉分析能力，但必须纳入门禁体系。

#### 运行方式

- 2~3 个视觉模型并发
- 1 套规则引擎并行
- 1 个共识/冲突收敛器

#### 输出约束

- 输出仅作为 `design_review_candidates`
- 不允许直接升格为 T1
- 默认最高 T2

---

### 6.4 Persona Sandbox

#### 保留原则

保留多 Persona、多模型 fan-out，但重新定义为：

> **假设压力测试器**

#### 运行方式

- Persona 模板驱动
- 多模型 fan-out
- 聚类与分歧归纳

#### 输出约束

- 所有内容强制标记为“模拟生成”
- 只能进入假设包、风险说明或下一步研究建议
- 默认最高 T2

---

### 6.5 Judgment Synthesizer

#### 职责

- 收敛证据、Vision、Persona 等中间结果
- 区分事实 / 推断 / 模拟 / 缺口
- 产出结构化结论与候选输出

#### 注意事项

- 不是自由总结器
- 必须受 Gate 约束

---

### 6.6 Output Router / Gate Evaluator

#### 输出类型

- `judgment_card`
- `light_report`
- `evidence_report`
- `design_review_report`
- `hypothesis_pack`

#### 路由规则

- `RQ0-RQ1`
  - 允许：判断卡片、假设包
- `RQ2`
  - 允许：轻量报告、设计预评估
- `RQ3` 且 T1 充分
  - 允许：证据型报告

#### 降权规则

- Vision 输出：最高 T2
- Persona 输出：最高 T2
- 外部行业资料：原则上 T2
- 真实业务数据 / 真实研究数据：T1

## 9. 状态对象设计

建议统一定义 `ResearchTaskState`，作为编排与恢复的共享状态：

```ts
type ResearchTaskState = {
  taskId: string;
  inputType: 'text' | 'design' | 'mixed';
  originalQuery: string;
  uploadedFiles: FileRef[];
  uploadedDesigns: FileRef[];

  enabledModules: {
    evidence: boolean;
    visionMoE: boolean;
    personaSandbox: boolean;
    externalSearch: boolean;
    multiModelReview: boolean;
  };

  subQuestions: SubQuestion[];
  evidencePool: EvidenceItem[];
  evidenceConflicts: EvidenceConflict[];
  visionFindings: VisionFinding[];
  personaFindings: PersonaFinding[];

  rqLevel: 'RQ0' | 'RQ1' | 'RQ2' | 'RQ3';
  tierCoverage: {
    tier1Count: number;
    tier2Count: number;
    tier3Count: number;
  };

  candidateOutputs: CandidateOutput[];
  finalOutputs: FinalOutput[];
  reviewStatus: 'draft' | 'pending_review' | 'approved' | 'rejected';

  runStats: {
    startedAt: string;
    finishedAt?: string;
    cost?: number;
    latencyMs?: number;
    warnings: string[];
  };
};
```

## 12. API 设计

### 12.1 任务 API

#### `POST /api/research/tasks`

创建任务。

#### `POST /api/research/tasks/{taskId}/preview-plan`

预览拆题数、支路数、预计耗时与成本档位。

#### `POST /api/research/tasks/{taskId}/run`

启动任务。

#### `GET /api/research/tasks/{taskId}`

获取任务详情。

#### `GET /api/research/tasks/{taskId}/stream`

SSE 推送任务实时事件。

#### `POST /api/research/tasks/{taskId}/cancel`

取消任务。

---

### 12.2 子问题与证据 API

#### `GET /api/research/tasks/{taskId}/sub-questions`

获取子问题列表。

#### `GET /api/research/tasks/{taskId}/evidence`

获取证据池，支持按 Tier / 来源 / 冲突筛选。

#### `POST /api/research/evidence/{evidenceId}/review`

审核证据，支持：

- 接受
- 降级
- 驳回

---

### 12.3 Vision / Persona API

#### `GET /api/research/tasks/{taskId}/vision`

获取 Vision 结果。

#### `POST /api/research/tasks/{taskId}/vision/rerun`

重跑 Vision 支路。

#### `GET /api/research/tasks/{taskId}/persona`

获取 Persona 输出。

#### `POST /api/research/tasks/{taskId}/persona/rerun`

重跑 Persona 支路。

---

### 12.4 报告 API

#### `GET /api/research/tasks/{taskId}/outputs`

获取候选输出。

#### `POST /api/research/tasks/{taskId}/reports/generate`

基于候选输出生成正式报告。

#### `GET /api/research/reports/{reportId}`

获取报告详情。

#### `POST /api/research/reports/{reportId}/review`

审核报告。

#### `POST /api/research/reports/{reportId}/export`

导出报告。

---

### 12.5 系统与观测 API

#### `GET /api/system/models`

获取模型清单。

#### `GET /api/system/model-policies`

获取模型策略。

#### `GET /api/system/observability/summary`

获取系统观测汇总。

- `queued`

- `partial_failed`
- `awaiting_review`
- `completed`
- `failed`

### 18.3 中断恢复

依赖 `task_checkpoints` 恢复：

- 当前节点
- 已完成支路
- 中间产物

## 20. 复杂性转移账本

### 20.1 多模型 fan-out

- 原复杂性位置：单模型 Prompt
- 新位置：模型路由层 + 收敛层
- 收益：覆盖面更强
- 新成本：冲突处理、成本、时延
- 失效模式：意见过多、无法收敛

### 20.2 Vision MoE

- 原复杂性位置：单模型视觉分析
- 新位置：视觉共识抽取器 + 规则引擎
- 收益：视角更全面
- 新成本：门禁复杂
- 失效模式：把启发式判断误当事实

### 20.3 Persona Sandbox

- 原复杂性位置：人工脑暴
- 新位置：模拟层 + 标签门禁层
- 收益：更快暴露风险与分歧
- 新成本：极易越权
- 失效模式：模拟观点冒充真实用户声音

## 21. 冻结边界（Gate Boundary）

当前建议冻结以下边界，不允许随意改写：

- Vision 输出不得升为 T1
- Persona 输出不得升为 T1
- 证据型报告必须要求 `RQ3 + T1充分`
- 前端不得持正式模型 Key
- Gate 逻辑必须由服务端统一判定

## 23. 主要风险与止损策略

### 风险 1：多模型冲突过多

**止损：**

- 前端显式展示“共识 / 冲突 / 缺口”
- 不强行统一结论

### 风险 2：Vision / Persona 越权

**止损：**

- 最高 T2
- 强制标签
- Gate 中心化判定

### 风险 3：成本与时延失控

**止损：**

- 支路可降级
- 默认主模型 + 复核模型，而非无限 fan-out
- 预览阶段提示预计成本 / 耗时

### 风险 4：前端过于复杂

**止损：**

- 默认先显示任务工作台和判断卡片
- 证据、Vision、Persona 放入高级视图

### 风险 5：双真相

**止损：**

- 文档、Gate 规则、状态机、前端标签统一从服务端事实源生成

## 25. 本文档结论

这套系统完全可以在保留高级能力的前提下落地，但前提是：

- 把 `modelClients.ts` 作为模型适配层，而不是产品主逻辑层
- 把多 Agent / 多模型能力全部纳入统一编排
- 把 Vision / Persona 结果明确降权与标注
- 把前端设计成研究工作台，而不是聊天框
- 把输出统一交给 Gate 决定

最终目标不是“调用更多模型”，而是：

> **让复杂能力在一个可追溯、可治理、可审阅的系统里稳定运行。**
