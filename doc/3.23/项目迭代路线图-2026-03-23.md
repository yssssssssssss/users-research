# 项目后续迭代优先级路线图

日期：2026-03-23

---

## 1. 路线图原则

当前项目已经能跑，但还不是生产级。  
后续迭代不应再追求“堆更多功能”，而应按下面顺序推进：

1. **先稳**
2. **再真**
3. **再强**
4. **最后做规模化**

---

## 2. P0：稳定性与可用性（最高优先级）

目标：让项目“持续可跑、可恢复、可复现”。

### P0-1. 启动链路稳定化

- 固化 Node 版本（建议 `.nvmrc`）
- 固化 `better-sqlite3` 兼容策略
- 保证一键启动 / 一键停止脚本稳定
- 降低 `tsx watch` 带来的后台残留问题

### P0-2. 持久化链路稳固

- 继续只保留一个 SQLite 真路径
- 明确数据库 schema 版本管理
- 增加启动时 DB 健康检查

### P0-3. 编排超时与失败治理

- 节点级 timeout 可配置
- 失败时输出更结构化错误
- 统一 fallback 标记
- 避免“任务卡死但前端无感知”

### P0-4. 实时流治理

- SSE 去重
- 只推必要事件
- 断线重连策略
- 前端事件时间线更可读

### P0 预期产出

- 本地启动稳定
- 任务恢复稳定
- 错误可定位
- 演示不掉链子

---

## 3. P1：真实性与证据可信度（第二优先级）

目标：让结果不只是“能看”，而是“更可信”。

### P1-1. 外部证据升级

- 从“搜索线索”升级到“原文抓取 + 摘要”
- 区分：
  - 搜索结果
  - 抓取原文
  - 已核查证据
- 支持来源去重、发布时间、出处留痕

### P1-2. 体验模型与真实证据严格分层

- 在 UI 上更明确区分：
  - 方法论框架
  - 模拟结果
  - 真实证据
- 防止用户误把框架证据当事实证据

### P1-3. Persona 边界更明确

- Persona 模块只作为假设发散
- 不再让其在结果页中显得“像真实结论”
- 更明确地挂到 hypothesis / risk 区

### P1-4. Vision 真实性增强

- 加强真实设计图输入验证
- 对无图情况只输出非常保守结论
- 增加图像来源标记与引用信息

### P1 预期产出

- 结果来源更清楚
- 边界更清楚
- 报告可信度更高

---

## 4. P2：模型能力与编排质量（第三优先级）

目标：让系统输出更稳、更准、更有层次。

### P2-1. 模型策略优化

- 梳理每个节点的最优模型
- 降低无效多模型调用
- 提高 fallback 质量
- 增加成本 / 时延监控

### P2-2. 多模型复核重构

- 当前复核容易超时
- 改成：
  - 轻量复核
  - 异步复核
  - 分层复核

### P2-3. 编排节点结果标准化

- 子问题格式更稳
- Judgment 输出结构更稳
- Vision / Persona 输出字段更统一

### P2-4. 体验模型路由优化

- 提高模型命中精度
- 降低 PDF 提取依赖
- 增强手动覆盖体验

### P2 预期产出

- 节点波动更小
- 输出质量更均衡
- 成本与时延更可控

---

## 5. P3：产品化与协作能力（第四优先级）

目标：从“能跑原型”走向“团队可用工具”。

### P3-1. 任务管理升级

- 任务筛选
- 任务搜索
- 任务标签
- 任务复制 / 重跑

### P3-2. 报告能力升级

- 报告版本树
- markdown / html / pdf 导出
- 审核记录留痕
- 报告差异对比

### P3-3. 证据治理升级

- 证据编辑
- 证据批量审核
- 证据来源追踪
- 证据引用关系图

### P3-4. 权限与协作

- 用户体系
- 角色权限
- 任务共享
- 操作审计

### P3 预期产出

- 团队内部可协作
- 可持续沉淀任务资产
- 更像正式产品后台

---

## 6. P4：生产化与规模化（最后优先级）

目标：具备更正式的线上能力。

### P4-1. PostgreSQL 正式化

- Prisma 路径与 SQLite 路径收敛
- 单一持久化抽象
- 迁移脚本标准化

### P4-2. 任务队列化

- runTask 异步化
- 队列与 worker 解耦
- 支持重试与优先级

### P4-3. 观测与报警

- 节点耗时
- 失败率
- fallback 率
- 报警与日志聚合

### P4-4. 部署体系

- dev / staging / prod 分层
- 配置管理
- CI/CD
- 回滚策略

### P4 预期产出

- 可线上部署
- 可运维
- 可扩展

---

## 7. 建议的实际推进顺序

### 未来 1 周

- 固化 Node 版本
- 稳定启动脚本
- 稳定 SQLite 路径
- 继续清理 dev 模式抖动

### 未来 2~3 周

- 提升外部证据真实性
- 明确 Persona / Vision / Experience Model 边界
- 改造多模型复核

### 未来 1~2 月

- 强化任务管理
- 强化报告版本化
- 强化证据治理
- 初步引入 PostgreSQL 正式持久化

### 未来更长周期

- 队列化
- 权限体系
- 生产部署
- 监控告警

---

## 8. 最后结论

这个项目下一步最应该做的，不是继续加新页面，而是按下面的顺序推进：

> **先解决稳定性，再解决真实性，再提升质量，最后再谈规模化。**

一句话版本：

> **P0 让它不掉；P1 让它不假；P2 让它更准；P3 让团队能用；P4 才是生产化。**
