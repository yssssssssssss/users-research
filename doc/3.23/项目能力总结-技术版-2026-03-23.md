# 项目能力总结（技术版）

日期：2026-03-23

---

## 1. 系统定位

当前系统是一个 **任务状态驱动的 AI Research Orchestrator**。

核心状态对象：

- `ResearchTaskState`

系统围绕这个状态做：

- 创建
- 编排
- 更新
- 持久化
- 回放
- 审核

它不是“请求-响应式 AI 页面”，而是“任务生命周期系统”。

---

## 2. 技术架构

### 前端

- React 19
- Vite
- Ant Design
- Zustand
- React Router

职责：

- 任务创建
- 历史任务恢复
- 当前任务工作台
- 证据 / Vision / Persona / Result / Report / Ops 页面
- SSE 实时事件展示

### 后端

- Fastify
- TypeScript
- 编排逻辑在 `orchestrator`
- 业务聚合在 `taskService`

职责：

- API
- 任务生命周期管理
- 编排执行
- 证据门禁
- 报告生成
- 审核流转
- SQLite / PostgreSQL / memory 持久化

### 数据层

- 当前本地主用：SQLite
- 未来兼容：PostgreSQL + Prisma

---

## 3. 核心模块拆解

### A. taskService

系统主服务层，负责：

- 创建任务
- 运行任务
- 获取任务摘要
- 获取任务完整状态
- 获取输出
- 生成报告
- 审核报告
- 审核证据
- rerun Vision / Persona
- 持久化选择

这是当前最核心的业务聚合层。

### B. textNodes / orchestrator

编排节点实现，包括：

- `executeProblemDecomposer`
- `executeExperienceModelAnalysis`
- `executeExternalSearch`
- `executeVisionMoE`
- `executePersonaSandbox`
- `executeJudgmentSynthesizer`
- `enrichTaskForRun`

这是系统的推理流水线。

### C. experienceModelService

负责：

- 体验模型库
- PDF 文本抽取
- 自动路由
- 手动覆盖
- 模型说明与问题模板生成

### D. modelGateway

负责：

- 模型客户端初始化
- 各节点模型路由
- fallback
- 多模型复核

### E. sqliteStore

负责：

- `task_snapshots`
- `reports`
- SQLite schema 初始化
- 任务 / 报告读写

## 4. 任务状态机

主要状态：

- `draft`
- `queued`
- `running`
- `partial_failed`
- `awaiting_review`
- `completed`
- `cancelled`
- `failed`

审核状态：

- `not_required`
- `pending`
- `approved`
- `rejected`
- `rework_required`

这套状态足够支撑：

- 运行中追踪
- 结果待审
- 证据重算
- 报告审批

---

## 5. 研究主流程

主流程大致是：

1. 创建 `ResearchTaskState`
2. `runTask`
3. Problem Decomposer
4. Experience Model Analysis
5. External Search（可选）
6. Vision MoE（可选）
7. Persona Sandbox（可选）
8. Judgment Synthesizer
9. 输出候选结果
10. 生成报告
11. 审核 / 复核 / 重算
12. 持久化

特点：

- 每阶段支持 checkpoint
- 支持中间状态保存
- 支持最终状态回放

## 6. 输出与门禁设计

输出类型：

- `judgment_card`
- `light_report`
- `evidence_report`
- `design_review_report`
- `hypothesis_pack`

---

## 7. 数据模型设计评价

优点：

- `ResearchTaskState` 统一
- 业务对象完整
- 前后端共享类型
- 任务 / 证据 / 输出 / 报告边界清晰

问题：

- `state_json` 快照便于原型，但复杂查询能力有限
- SQLite 与 Prisma/Postgres 为两套持久化路径
- 当前事件流是快照驱动，不是细粒度 event sourcing

结论：

- 对原型阶段合适
- 对生产阶段仍需进一步拆分

---

## 8. 当前真实能力 vs 模拟能力

### 真实能力

- 真实任务创建
- 真实状态推进
- 真实模型调用
- 真实外部搜索接入
- 真实报告生成
- 真实 SQLite 落库
- 真实历史任务恢复

### 半真实 / 辅助能力

- Vision：有图时较真实，无图时弱推断
- Persona：永远模拟
- Experience Model：方法框架，不是事实
- External Search：目前主要是待核查证据池，不是全文证据验证

### 明确 fallback 能力

- 文本模型失败可退回本地 fallback
- Vision / Judgment / Search 都有降级逻辑
- 系统目标是“别卡死”，不是“绝不降级”

---

## 9. 当前主要优点

### 优点 1：链路完整

不是 PPT 架构，是真能跑。

### 优点 2：状态模型统一

任务是中心，其他都是派生视图。

### 优点 3：门禁意识强

没有把 Persona / Vision 伪装成高等级事实证据。

### 优点 4：可持久化

SQLite 让本地闭环成立了。

### 优点 5：前后端已形成工作台

已经是产品雏形，不只是 API demo。

---

## 10. 当前主要问题

### 问题 1：dev 启动链路脆弱

- Node 版本切换会影响 `better-sqlite3`
- `tsx watch` 后台化稳定性一般

### 问题 2：真实证据链还不够强

- 外部搜索更多是线索，不是事实级验证
- 体验模型与 Persona 容易被误解为“证据”

### 问题 3：模型超时明显

- `multiModelReview` 可能超时
- 复杂节点时延偏高

### 问题 4：持久化策略仍偏原型

- SQLite 快照可用
- 但还没有更强的数据治理方案

### 问题 5：前端仍偏展示型

- 更像工作台原型
- 不是成熟运营后台
