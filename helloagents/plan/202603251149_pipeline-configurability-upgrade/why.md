# 变更提案: 分析链路可配置化升级

## 需求背景
当前项目四条关键能力链路仍有明显硬编码：
- 外部搜索引擎优先级与结果来源筛选写死在服务端；
- 视觉评审的角色与 system prompt 写死在 `VISUAL_REVIEWERS`；
- Persona 数字人主要依赖内置画像模板，再由模型二次润色；
- 体验模型 PDF 仅做直接抽取与缓存，没有文档索引层，更没有向量化能力。

这会导致三个实际问题：
1. 运营 / 研究人员无法热更新搜索源白名单、禁用站点或调整检索策略；
2. Prompt 无法按任务类型和用户目标动态匹配，只能靠改代码；
3. Persona 与体验模型都缺少“文档化、可治理、可扩展”的知识底座。

## 产品分析

### 目标用户与场景
- **用户群体:** 内部研究人员、产品经理、策略运营、系统维护者
- **使用场景:** 调整分析链路策略、接入新的用户画像文档、升级评估提示词、增强体验模型资料利用方式
- **核心痛点:** 现在很多能力是“可运行但不可运营”；想改规则只能改代码、发版、重跑

### 价值主张与成功指标
- **价值主张:** 把当前原型链路升级为“配置驱动的分析平台骨架”
- **成功指标:** 
  - 外部源策略可在不改代码前提下调整并生效
  - 视觉 prompt 可按任务上下文动态选择
  - Persona 可从用户研究文档生成
  - 体验模型资料从“直接读 PDF”升级为“可索引、可追踪”的资料层

### 人文关怀
配置能力开放后，必须保留审计、版本与默认降级机制，避免运营误配导致错误分析结论扩散。

## 变更内容
1. 设计外部源治理层：搜索引擎策略、域名白名单 / 黑名单、来源元数据、热更新机制。
2. 设计视觉 prompt 注册表：多 prompt 模板库、标签化匹配、默认回退。
3. 设计 Persona 知识底座：从用研报告 / 用户画像文档抽取结构化 persona，再驱动数字人生成。
4. 设计体验模型资料索引层：保留 PDF 抽取缓存，同时引入 chunk / metadata / 可选向量化方案。

## 影响范围
- **模块:** `apps/server`, `apps/web`, `packages/shared`, `packages/model-clients`
- **文件:** `searchClient.ts`, `textNodes.ts`, `visualReview.ts`, `roles/personas.ts`, `experienceModelService.ts`, system / ops 相关接口与页面
- **API:** 新增系统配置读取 / 更新接口；可选新增 Persona 文档资料接口
- **数据:** 新增配置表 / JSON 存储、Persona 文档索引、体验模型文档索引

## 核心场景

### 需求: 外部源策略热更新
**模块:** apps/server
允许维护者配置搜索引擎顺序、白名单 / 黑名单域名、来源信誉标签，并在不改代码的情况下生效。

#### 场景: 运营禁用低质量站点
已有搜索策略配置时
- 更新域名策略后，新任务检索必须按新规则筛选
- 历史任务保留旧 trace，不被重写

### 需求: 视觉 prompt 库动态匹配
**模块:** apps/server
把视觉角色 prompt 从静态常量改成可注册模板库。

#### 场景: 针对不同任务类型切换 prompt
运行视觉评审时
- 系统按任务类型 / 用户目标 / 画面类型选择最合适模板
- 未命中时回退到默认模板

### 需求: Persona 基于文档生成
**模块:** apps/server
支持从上传或预置的用研报告、用户画像文档提取 persona 素材。

#### 场景: 基于画像文档生成数字人
存在用研文档时
- 系统先抽取结构化 persona
- 再驱动数字人评论与评分

### 需求: 体验模型资料索引升级
**模块:** apps/server
把体验模型 PDF 从“直接读全文”升级成“抽取 + chunk + metadata + 可选向量化”的资料层。

#### 场景: 模型按维度检索体验模型依据
执行体验模型评估时
- 系统应能定位到更细粒度章节
- 若未开启向量化，也能用 chunk/关键词索引回退

## 风险评估
- **风险:** 新增配置层会引入状态一致性与热更新复杂度
- **缓解:** 先做“默认配置 + 显式版本 + 热加载边界清晰”的最小方案，再决定是否加 UI 配置台
- **风险:** Persona 与体验模型资料升级容易过度设计
- **缓解:** 第一阶段先做结构化抽取与索引，不强行一次接入全量向量数据库
